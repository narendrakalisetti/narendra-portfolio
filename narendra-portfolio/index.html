⚡ Azure Engineering · PySpark Structured Streaming
Why I used a 10-minute watermark in the fraud detection pipeline — not 1 hour
March 20265 min readNarendra Kalisetti

When I built the real-time fraud detection pipeline on Azure Databricks, one of the first decisions I had to make was: what watermark threshold should I set on my PySpark Structured Streaming job?

Most tutorials default to 1 hour, or even longer. I went with 10 minutes. Here's exactly why — and why getting this wrong would have broken both our latency SLA and our MLR 2017 compliance guarantees.

What is a watermark, and why does it matter for fraud?

In PySpark Structured Streaming, a watermark defines how long the engine waits for late-arriving data before closing a time window. Set it too long and your state store balloons — and more critically, you delay your risk scoring output.

Our SLA: flag suspicious transactions in under 200ms end-to-end. Every second of watermark tolerance directly adds to state management overhead and output latency.

The MLR 2017 constraint that made 1 hour impossible

Under MLR 2017 Regulation 35, SAR flags must be raised and routed within the same micro-batch cycle as detection — you cannot retroactively close a window and raise a SAR an hour later. A 1-hour watermark creates an unacceptable compliance gap.

Why 10 minutes hits the sweet spot

  • 97.3% of transactions arrived within 90 seconds of their event timestamp
  • 99.2% arrived within 5 minutes
  • The remaining 0.8% were retry events handled by idempotency keys — not the watermark

The implementation

df_with_watermark = (
    df_raw
    .withWatermark("event_timestamp", "10 minutes")
    .groupBy(window("event_timestamp", "1 minute"), "account_id")
    .agg(
        count("transaction_id").alias("txn_count"),
        sum("amount_gbp").alias("total_amount"),
        countDistinct("merchant_country").alias("country_count")
    )
)
Takeaway: Always set your watermark based on measured late-arrival data from your actual source — not a tutorial default. For compliance-sensitive pipelines, the watermark directly affects your regulatory obligations.

Remote state locking in Azure: why two engineers pushing simultaneously broke our lakehouse
January 20266 min readNarendra Kalisetti

It was a Thursday afternoon. Two engineers ran terraform apply at almost exactly the same time. Within minutes — corrupted state file, partially provisioned workspace, lakehouse down for 3 hours.

What went wrong: the race condition

By default, Terraform stores state locally. Both engineers read the old state independently, planned against it, then both attempted to write new state simultaneously — a split-brain where real infrastructure and the state file diverged.

The first engineer provisioned the ADLS Gen2 storage account. The second engineer's apply overwrote state as if it had never been created. The next plan tried to create it again — which failed because it already existed in Azure.

The fix: Azure Blob remote state with lease-based locking

terraform {
  backend "azurerm" {
    resource_group_name  = "rg-terraform-state"
    storage_account_name = "energyzerotfstate"
    container_name       = "tfstate"
    key                  = "energyzero/prod/terraform.tfstate"
    use_azuread_auth     = true
  }
}

use_azuread_auth = true authenticates via Azure AD — no shared storage keys, full RBAC control over who can read/write state.

What happens now

Engineer A acquires the blob lease. Engineer B sees: "Error acquiring the state lock". No race condition. No split-brain. No corrupted state.

Takeaway: Remote state with blob locking is not optional on any team workflow. Bootstrap it on day one. The 20-minute setup saves hours of incident recovery.

Implementing MLR 2017 SAR routing in a PySpark Streaming pipeline — what the docs don't tell you
February 20267 min readNarendra Kalisetti

MLR 2017 is well-documented in legal terms. What's not documented is how to implement it in a real-time data pipeline. Regulation 35 requires SARs filed with the NCA — but says nothing about Kafka, Delta Lake, or idempotency.

Decision 1: SAR routing inside foreachBatch, not a separate job

A separate scheduled job violates the consent SAR timing requirement. SAR routing must sit in the same pipeline stage as fraud scoring:

def process_batch(batch_df, batch_id):
    scored_df = apply_risk_scoring(batch_df)
    sar_candidates = scored_df.filter(
        (col("amount_gbp") > 5000) & (col("risk_score") > SAR_THRESHOLD)
    )
    if sar_candidates.count() > 0:
        route_to_sar_queue(sar_candidates, batch_id)
    scored_df.write.format("delta").mode("append").save(SILVER_PATH)

Decision 2: Idempotency keys to prevent duplicate SARs

df_with_key = df.withColumn(
    "sar_idempotency_key",
    sha2(concat(col("transaction_id"), lit(str(batch_id))), 256)
)

Decision 3: Resolving the GDPR / MLR tension

Pseudonymise using SHA-256 with a rotating salt in Key Vault. For the SAR path, pass an encrypted reference key plus a secure lookup table — so the NCA can request decryption via formal legal process, but raw PII never touches the streaming state store.

Takeaway: MLR 2017 SAR routing belongs in your streaming write path. Build idempotency from day one. Design the GDPR/MLR tension as a data architecture decision.
